## 추가 작업

### Firebase 공지 화면
- **관리자-공지 등록**: 공지사항 작성, 등록, 수정, 삭제 화면
- **사용자-공지 조회**: 공지사항 목록 조회, 상세 조회 화면
- **기술 스택**: Firebase Firestore
- **담당자**: 추가 작업 (담당자 배정 필요)

### Firebase 상담 채팅
- **관리자-상담 채팅**: 고객 상담 채팅 화면 (채팅방 목록, 메시지 송수신)
- **사용자-상담 채팅**: 고객 상담 채팅 화면 (채팅방 생성, 메시지 송수신)
- **기술 스택**: Firebase Firestore (실시간 채팅)
- **담당자**: 추가 작업 (담당자 배정 필요)

## 🎯 분배 원칙

### 1. 로직 유사성 고려
- **담당자 1**: 로그인과 개인정보 수정은 사용자/관리자 모두 비슷한 로직 (단순 CRUD)
  - 관리자 화면은 태블릿용이지만 로직은 동일
- **담당자 2**: 상품 조회와 상세 화면은 연관성이 높아 한 명이 담당
- **담당자 3**: 장바구니, 주문, 결제까지의 프로세스는 하나의 연속된 흐름으로 한 명이 담당

### 2. 기술적 특징
- **장바구니**: DB 테이블 없이 임시 저장(메모리/로컬스토리지)만 사용
- **주문 내역 조회**: 결제 프로세스와 연관성이 높아 담당자 3이 담당

### 3. 작업량 균형
- 화면 수: 2-6개 (평균 3.8개)
- 담당자 3의 작업량이 다소 많지만, 하나의 연속된 프로세스이므로 효율적

## 📝 주요 결정 사항

1. ✅ **로그인/개인정보 수정 통합**: 사용자와 관리자 모두 담당자 1이 담당 (비슷한 로직)
2. ✅ **상품 조회 통합**: 상품 조회와 상세 화면은 담당자 2가 담당 (연관성 높음)
3. ✅ **주문/결제 프로세스 통합**: 장바구니, 주문, 결제까지 담당자 3이 담당 (한 흐름)
4. ✅ **장바구니**: DB 테이블 없이 임시 저장만 사용
5. ✅ **주문/반품 조회 분리**: 사용자용(담당자 4)과 관리자용(담당자 5)으로 분리
   - 담당자 4: 사용자 주문 조회 + 사용자 수령/반품 조회
   - 담당자 5: 관리자 주문 관리 + 관리자 수령/반품 관리
6. ✅ **PM 및 3D 프리뷰**: 담당자 6이 프로젝트 관리 및 상품 상세 화면 3D 프리뷰 기능 개발 담당
7. ✅ **추가 작업**: Firebase 공지 화면 (관리자 등록, 사용자 조회) 및 Firebase 상담 채팅 (관리자, 사용자) 화면 개발 필요
