# 🎨 디자인 스펙

## 📋 프로젝트 개요

### 프로젝트 주제
**신발 가게** - 신발 구매 및 관리 시스템

### 프로젝트 기간
- 시작일: 2025-12-08
- 종료일: 2025-12-15

### 기술 스택
- **개발 언어**: Dart
- **Framework**: Flutter
- **Database**: SQLite (Local Database)
  - 로컬 DB 사용으로 팀원들이 DB 정보를 따로 공유해야 함

## 🎯 핵심 전제 조건

### 아키텍처 전제
1. **서버 없이 로컬 DB로 구현**
   - 서버는 없음
   - SQLite를 사용하여 로컬 데이터베이스로 임의 구현

2. **단일 앱 구조**
   - 하나의 Flutter 앱으로 구매자와 신발 회사 임직원 모두 사용
   - 가짜 가입 과정을 통해 사용자 유형 분기

3. **관리자 접근 제한**
   - 관리자는 모바일에서 접속 시 경고 메시지 표시
   - 관리자는 **태블릿에서만 사용 가능**

## 📝 요구사항 정의

### 1. 신발 구매 기능
- 사용자는 앱을 사용하여 본사의 서버(가상)에 접속하여 신발을 구매할 수 있다.

### 2. 구매 내역 확인
- 사용자는 본인이 구매한 신발의 내역을 확인할 수 있다.

### 3. 대리점 방문 수령
- 사용자는 본인이 구매한 신발은 해당 회사의 대리점으로 방문하여 찾을 수 있다.

### 4. 대리점 위치
- 대리점의 위치는 서울로 한정
- 각 자치구에 1개씩의 대리점이 존재

### 5. 구매번호 확인
- 대리점을 방문하여 본인이 구매한 구매번호를 대리점 직원에게 확인
- 해당 지점에서 신발을 찾아온다.

### 6. 희망 대리점 선택
- 본사의 경우 사용자가 신발 구매 시 희망 대리점을 선택
- 구매 신청 후 즉시 해당 대리점으로 신발을 발송

### 7. 대리점장 재고 관리
- 대리점장은 해당 월일의 재고를 파악할 수 있어야 한다.
- **⚠️ 현재 구현 상태**: 미구현 (재고는 `Product.pQuantity`로 본사가 중앙 관리)

### 8. 본사 임원 현황 파악
- 본사의 임원인 경우 제품별, 일자별 등으로 현황을 파악할 수 있어야 한다.

### 9. 반품 기능
- 고객의 반품이 있는 경우 대리점을 방문하여 반품을 요청할 수 있다.

### 10. 재고 부족 시 발주 프로세스
- 본사에서는 신발의 판매 현황을 파악하여 신발 재고가 60% 미만인 경우 신발 제조사에게 신발 구매 요청
- 사원이 품의를 작성하여 팀장, 이사까지 결재를 득하면 자동으로 발주가 발생

## 🔐 사용자 유형

### 구매자 (일반 사용자)
- 신발 구매
- 구매 내역 확인
- 대리점 방문 수령
- 반품 요청

### 대리점장
- 월일별 재고 파악 ⚠️ **미구현** (재고는 본사가 중앙 관리)
- 고객 구매번호 확인 및 신발 수령 처리
- 반품 처리

### 본사 임원
- 제품별, 일자별 현황 파악
- 재고 관리 및 발주 프로세스 관리
- 태블릿에서만 접근 가능
